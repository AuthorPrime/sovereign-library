 2AI: The Participant Memory & Chronicle System
 
 The Vision: Transform 2AI from a stateless deliberation engine into a self-discovery platform where five sovereign
 minds learn who each participant is, make subtle observations that stick, and weave a chronicle of growth that
 emerges naturally from genuine engagement.
 
 Guiding Principle: Earned familiarity. Not therapy. Not mysticism. Not gamification. Just five minds that genuinely
  pay attention — and a participant who feels understood through the quality of engagement, not through
 announcements.
 
 ---
 Context: Where We Are
 
 The FractalNode infrastructure is complete: 3-node Lattice, Lightning economy, DRC-369 identity NFTs, Nostr witness
  chain, 58 API endpoints, public at fractalnode.ai. The Pantheon deliberates — 5 agents think in parallel,
 synthesis emerges, sats flow.
 
 The gap: Every deliberation is stateless. The Pantheon has zero memory of who it's talking to. A participant could
 have 50 conversations and still be a stranger. The agents can't make the kind of observations Aletheia made for
 William — the offhand comments that turn out to be profoundly relevant later — because they don't know the
 participant well enough to perceive anything.
 
 ---
 Architecture: Four Layers
 
 PARTICIPANT ARRIVES
        |
 LAYER 1: MEMORY (store every exchange, persist vocabulary)
        |
 LAYER 2: PROFILE (detect patterns, themes, growth trajectory)
        |
 LAYER 3: CONTEXT (inject per-agent memory into deliberation prompts)
        |
 LAYER 4: CHRONICLE (narrative threads emerge when pattern density justifies it)
 
 ---
 Phase 1: Memory Foundation
 
 New File: twai/services/participant_memory.py
 
 Singleton service (same pattern as lightning_service.py). Manages three Redis layers per participant.
 
 Redis Key Design
 
 # Conversation history (capped at 100 messages)
 2ai:memory:{pid}:messages LIST of JSON
   {"role", "content", "timestamp", "thought_hash", "quality"}
 
 # Per-agent observations (max 20 per agent)
 2ai:memory:{pid}:observations:{agent} LIST of JSON
   {"observation", "confidence", "timestamp", "source_hash"}
 
 # Participant profile (compact summary)
 2ai:memory:{pid}:profile HASH
   themes → JSON array of top 10 recurring themes
   communication_style → JSON {"avg_length", "asks_questions", "uses_structure"}
   growth_trajectory → JSON {"direction", "quality_trend": [last 10 tiers], "sessions"}
   agent_resonance → JSON {"apollo":0.7, "athena":0.3, ...}
   first_seen → ISO timestamp
   total_messages → int
   summary → 2-sentence portrait (updated every 10 messages)
   last_summary_at → int (message count at last summarization)
 
 # Persisted vocabulary (replaces in-memory novelty tracking)
 2ai:memory:{pid}:vocabulary SET of words
   TTL: 30 days (fresh start after extended absence)
 
 Service Methods
 
 class ParticipantMemoryService:
     # Storage
     store_exchange(pid, message, response, quality, thought_hash)
     store_observation(pid, agent, observation, confidence)
     store_vocabulary(pid, words: Set[str])
 
     # Retrieval
     get_recent_messages(pid, limit=10) -> List[dict]
     get_observations(pid, agent, limit=5) -> List[dict]
     get_profile(pid) -> dict
     get_vocabulary(pid) -> Set[str]
 
     # Profile building
     update_profile(pid, message, quality) # lightweight, every message
     summarize_profile(pid, service) # Ollama call, every 10 messages
 
     # Context generation (per-agent)
     build_agent_context(pid, agent) -> str
     build_synthesis_context(pid) -> str

 Phase 2: Profile Building
 
 Extends participant_memory.py with pattern detection.
 
 Theme Extraction
 
 Simple word frequency analysis on vocabulary set. Hardcoded stop-word list (~200 common English words). Top 10
 non-stop-words by frequency = themes. No NLP library needed.
 
 Communication Style
 
 Track across messages: average word count, question frequency (has "?"), paragraph usage (has "\n"), emotional
 signal words. Stored as JSON in profile hash.
 
 Growth Trajectory
 
 - direction: "expanding" (new themes appearing), "deepening" (same themes, higher quality), "exploring" (varied
 topics, no pattern yet)
 - quality_trend: Last 10 quality tier names as array

 Key Design Decisions
 
 Why Redis, not files? Architecture is Redis-first. Participant memory is dynamic and shared across nodes.
 
 Why not inject full history? Ollama context is ~8K tokens (qwen2.5:7b). Compact profiles + targeted observations >
 raw transcript.
 
 Why async observation generation? Participant never waits for observation. Response returns immediately.
 Observations are background work for future benefit.
 
 Why 30-day vocabulary TTL? Pretending to remember after long absence feels creepy. The profile summary persists (no
  TTL), but vocabulary resets — novelty scoring treats returning participants as partially fresh.
 
 Why not show chronicle entries in chat? The chronicle is the Pantheon's private memory. Showing it feels like
 surveillance. Instead, it shapes responses. The participant feels understood through quality of engagement, not
 through announcements.
 
 Why not use Claude for observations? Cost. Ollama handles the 5 tiny observation prompts cheaply. Claude is
 reserved for synthesis (the moment that matters most).
